## 新增需求

### 需求：人员管理表格（列表）

系统必须在 `/team` 页面提供人员管理表格，用于展示人员列表并支持进入新增/编辑/删除流程。

表格必须展示以下字段列：

- 姓名
- 性别
- 身份证号
- 职位
- 所属部门
- 职级

#### 场景：列表有数据

- **当** 用户进入 `/team` 页面且人员列表存在数据
- **那么** 系统应在表格中逐行展示人员信息，并提供每行的“编辑/删除”操作入口

#### 场景：列表为空

- **当** 用户进入 `/team` 页面且人员列表为空
- **那么** 系统应展示空状态提示，并提供“新增人员”的入口

### 需求：新增人员（Create）

系统必须允许用户新增一名人员，并将其加入人员列表。

#### 场景：新增成功

- **当** 用户点击“新增人员”，输入合法的人员信息并提交保存
- **那么** 系统应创建人员记录，并在表格中可见该人员

#### 场景：身份证号重复

- **当** 用户新增人员时输入的身份证号已存在于人员列表中
- **那么** 系统应阻止保存并提示“身份证号已存在”（或等价错误信息）

### 需求：枚举字段约束

系统必须将性别、所属部门、职级作为固定枚举值进行管理，用户只能选择枚举中的值。

#### 场景：枚举字段选择

- **当** 用户在新增或编辑人员时填写性别/所属部门/职级
- **那么** 系统应提供可选项列表，并仅允许选择枚举中的值

### 需求：编辑人员（Update）

系统必须允许用户编辑人员信息，并将修改结果反映到人员列表中。

#### 场景：编辑成功

- **当** 用户对某条人员记录执行“编辑”，修改任意字段并保存
- **那么** 系统应更新该人员记录，并在表格中展示更新后的信息

#### 场景：编辑导致身份证号冲突

- **当** 用户编辑人员时将身份证号修改为已被其他人员占用的号码
- **那么** 系统应阻止保存并提示“身份证号已存在”（或等价错误信息）

### 需求：删除人员（Delete）

系统必须允许用户删除人员记录，并防止误删。

#### 场景：确认删除

- **当** 用户对某条人员记录执行“删除”并在确认提示中确认
- **那么** 系统应删除该人员记录，且该人员不再出现在表格中

#### 场景：取消删除

- **当** 用户对某条人员记录执行“删除”但在确认提示中取消
- **那么** 系统不应删除该人员记录

### 需求：Mock 数据来源（无后端）

系统必须在无后端依赖的情况下工作，人员数据来源为前端 mock。

#### 场景：无后端可用

- **当** 用户访问 `/team` 页面且不存在后端 API
- **那么** 系统应仍可完成新增/编辑/删除操作，并在当前会话中反映到表格中
